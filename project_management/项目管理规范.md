# OPC-Agents 项目管理规范

## 一、概述

本规范基于项目管理诊断报告，旨在建立OPC-Agents系统的项目管理流程，确保需求管理、范围控制和迭代流程的有效性。

## 二、需求管理

### 2.1 需求准入门槛

每个进入开发的需求必须先回答三个问题：

**问题1：解决了什么问题？**
- 用户/客户的哪个具体痛点
- 谁在使用这个功能
- 不用会怎样

**问题2：做完的标志是什么？**
- 交付边界是什么
- 明确不做什么

**问题3：优先级是什么？**
- P0（必须做）- 最多同时2个
- P1（应该做）- 尽快安排
- P2（可以等）- 进入 backlog

### 2.2 需求范围承诺书

每个需求必须填写《需求范围承诺书》，明确交付边界和不做的内容。

### 2.3 每周需求评审会

**时间**：每周一上午（30分钟）
**参与者**：CEO + 技术负责人 + 产品/PM
**议程**：
1. 回顾上周上线的功能是否达到预期（10分钟）
2. 评审新增需求（15分钟）
3. 确认本周开发范围（5分钟）

**关键规则**：CEO的需求和其他人一样，都需要经过评审才能进入开发。

## 三、迭代流程

### 3.1 迭代周期

- 迭代周期：1周
- 迭代目标：明确、可交付

### 3.2 每日站会

**时间**：每天早上（15分钟）
**议程**：
- 每人1分钟，只说：今天做什么 + 遇到什么问题
- 重点：谁说"卡住了"——这个人优先解决
- 不要汇报进度，要汇报障碍

### 3.3 上线流程

#### 上线前必须确认：

1. **原始问题解决了吗？**
2. **超出原始范围了吗？**
3. **有人测试过吗？**

如果任何一个答案不确定，这一周的工作就不能上线。

## 四、工具使用

### 4.1 推荐工具栈

| 用途 | 推荐工具 | 说明 |
|------|----------|------|
| 需求池 | Notion | 免费，建"需求池"数据库，加状态/优先级/提出人字段 |
| Sprint管理 | Notion Board | 看板：待评审 → 开发中 → 测试 → 上线 |
| 范围文档 | TRAE + 模板 | 在需求准入时强制填写范围承诺书 |

### 4.2 Notion需求池字段设计

```
需求名称 | 描述 | 提出人 | 提出日期 | 优先级(P0/P1/P2) | 状态 | 承诺交付 | 明确不做 | 评审结论
```

## 五、优先级排序

### 第一优先级（本周执行）
1. 建立"范围承诺书"模板，CEO需求优先适用
2. 设立每周一需求评审会（30分钟）
3. 上线前必须回答"上线三问"

### 第二优先级（第2周执行）
1. Notion需求池建立
2. 看板建立
3. 站会改为问题导向

### 第三优先级（第3-4周）
1. 复盘上周改进效果
2. 根据实际运行情况调整流程
3. 沉淀团队项目管理规范文档

## 六、预期效果

| 指标 | 改进前 | 改进后（预期） |
|------|--------|----------------|
| 单次迭代范围蔓延率 | 难以统计 | <15% |
| 需求评审通过率 | 100%（无评审） | 60-70% |
| 上线回滚率 | 高 | 显著降低 |
| 团队"救火"时间占比 | 高 | 明显减少 |

## 七、变更管理

### 7.1 需求变更流程

1. 需求变更必须提交新的《需求范围承诺书》
2. 变更需要重新评审
3. 变更影响当前迭代的，需要评估是否调整迭代范围

### 7.2 风险管理

- 每个需求必须识别潜在风险
- 风险等级：高、中、低
- 高风险需求需要制定风险缓解计划

## 八、文档管理

### 8.1 文档结构

- `project_management/`
  - `需求范围承诺书模板.md`
  - `项目管理规范.md`
  - `迭代计划/` - 每次迭代的计划和回顾
  - `需求文档/` - 已评审通过的需求文档

### 8.2 文档更新机制

- 需求文档在评审通过后更新
- 项目管理规范根据实际运行情况每季度更新
- 迭代回顾文档在每次迭代结束后更新

## 九、附则

本规范自发布之日起生效，由项目管理团队负责解释和修订。
