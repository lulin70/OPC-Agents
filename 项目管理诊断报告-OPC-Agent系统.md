# OPC Agent管理系统 - 项目管理诊断报告

**诊断时间**：2026年3月26日  
**团队规模**：5人以下  
**业务类型**：面向个人创业OPC的Agent管理系统  
**诊断维度**：需求管理、范围控制、迭代流程

---

## 一、问题诊断

### 1.1 核心问题：无门槛需求流入

**问题描述**：所有需求自动进入开发，没有任何准入机制。

**典型案例**：总裁办Agent → 三贤者战略评估Agents → 个人助理Agent

这是典型的**需求连锁反应**——一个需求触发另一个需求，没有边界控制。

### 1.2 根因分析

| 症状 | 根本原因 |
|------|----------|
| CEO拍脑袋就能改 | 没有需求评审机制，任何需求都默认"可以做" |
| TRAE文档没人看 | 有文档但没有与开发流程绑定，变成了"记录用的" |
| 1周迭代但范围蔓延 | 快节奏掩盖了失控——快速迭代优势被范围蔓延消耗 |
| 三贤者案例 | 需求没有"入口控制"，每个需求进来后自动触发新需求 |

### 1.3 风险评估

- **当前状态**：Red（高风险）
- **如果不改变**：团队会持续处于"永远做不完"的状态，核心功能无法稳定交付
- **改进难度**：低——不需要换人换工具，只需要加流程约束

## 二、改进方案

### 2.1 建立需求准入门槛

每个进入开发的需求必须先回答三个问题：

**问题1：解决了什么问题？**
- 用户/客户的哪个具体痛点
- 谁在使用这个功能
- 不用会怎样

**问题2：做完的标志是什么？**
- 交付边界是什么
- 明确不做什么

**问题3：优先级是什么？**
- P0（必须做）- 最多同时2个
- P1（应该做）- 尽快安排
- P2（可以等）- 进入 backlog

### 2.2 上线三问规则

每次上线前必须确认：

1. **原始问题解决了吗？**
2. **超出原始范围了吗？**
3. **有人测试过吗？**

如果任何一个答案不确定，这一周的工作就不能上线。宁可少做一个功能，也不能带着蔓延上线。

### 2.3 范围承诺书模板

每个需求必须填写以下内容：

```markdown
## 需求范围承诺书

**需求名称**：_______________
**提出日期**：_______________
**解决的问题**：_______________

**承诺交付**：
1. ...
2. ...

**明确不做**（这一条最重要）：
1. ...
2. ...

**蔓延触发器识别标准**：
如果有人提出"能不能顺便..."：
- 是否在"明确不做"清单里？→ 是 → 否决
- 是否需要单独的需求准入评审？→ 是 → 走新需求流程
```

### 2.4 每周需求评审会

**时间**：每周一上午（30分钟）  
**参与者**：CEO + 技术负责人 + 产品/PM  
**议程**：
1. 回顾上周上线的功能是否达到预期（10分钟）
2. 评审新增需求（15分钟）
3. 确认本周开发范围（5分钟）

**关键规则**：CEO的需求和其他人一样，都需要经过评审才能进入开发。

### 2.5 每日站会改革

**当前问题**：站会变成进度汇报，失去发现问题价值

**改进方案**：
- 每人1分钟，只说：今天做什么 + 遇到什么问题
- 重点：谁说"卡住了"——这个人优先解决
- 不要汇报进度，要汇报障碍

## 三、工具升级建议

### 3.1 推荐工具栈

| 用途 | 推荐工具 | 说明 |
|------|----------|------|
| 需求池 | Notion | 免费，建"需求池"数据库，加状态/优先级/提出人字段 |
| Sprint管理 | Notion Board | 看板：待评审 → 开发中 → 测试 → 上线 |
| 范围文档 | TRAE + 模板 | 在需求准入时强制填写范围承诺书 |

### 3.2 Notion需求池字段设计

```
需求名称 | 描述 | 提出人 | 提出日期 | 优先级(P0/P1/P2) | 状态 | 承诺交付 | 明确不做 | 评审结论
```

## 四、优先级排序

### 第一优先级（本周执行）
1. 建立"范围承诺书"模板，CEO需求优先适用
2. 设立每周一需求评审会（30分钟）
3. 上线前必须回答"上线三问"

### 第二优先级（第2周执行）
1. Notion需求池建立
2. 看板建立
3. 站会改为问题导向

### 第三优先级（第3-4周）
1. 复盘上周改进效果
2. 根据实际运行情况调整流程
3. 沉淀团队项目管理规范文档
